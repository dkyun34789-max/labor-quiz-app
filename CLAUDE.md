## 앱 구조 및 화면 플로우

화면 전환은 JS로 `.screen` 섹션에 `.active` 클래스를 토글하는 방식이다(`showScreen(name)` 단일 함수).
SPA 라우터나 history API는 사용하지 않는다.

```
[시작 화면]
  └ 이름·사번 입력 → "시작하기"
       ↓
[안내 화면]
  └ 카테고리 4개 순서 카드 → "퀴즈 시작 →"
       ↓
[문항 화면] × 20
  └ 보기 선택 → "정답 확인" → 인라인 피드백 → "다음 문항 →"
       ↓ (20번 제출 후 "결과 보기")
[결과 화면]
  └ 총점/정답률/카테고리별 결과/오답 토글
       ├ "다시 도전하기" → 시작 화면
       └ "순위 보기"     → saveResult() 후 리더보드 화면
            ↓
[리더보드 화면]
  └ 내 기록 요약 + 전체 순위 → "다시 도전하기" → 시작 화면
```

화면 식별자(섹션 id):
`screen-start` / `screen-intro` / `screen-quiz` / `screen-result` / `screen-leaderboard`

## 전역 상태 (state 객체)

```javascript
const state = {
  userName: '',        // 시작 화면에서 입력한 이름
  employeeId: '',      // 사번 (동명이인 구분용 식별자, Storage 키 기준)
  currentQ: 0,         // 현재 문항 인덱스 (0~19)
  answers: [],         // 사용자 선택 답안 인덱스 배열
  score: 0,            // 누적 정답 수
  wrongIds: [],        // 오답 문항 id 배열
  startTime: null,     // 퀴즈 시작 시각 (Date)
};
```

state는 전역 단일 객체로 관리한다. "다시 도전하기"는 위 초기값으로 리셋한다.

## 문항 데이터 구조

```javascript
// questions 배열 — JS 내부에 정적으로 정의
{
  id: 1,
  category: '근로시간',           // 4개 카테고리 중 하나
  type: 'multiple_choice',        // 'multiple_choice' | 'ox' | 'case'
  question: '문항 텍스트',         // 부정형 강조용 <strong> 태그 허용
  options: ['①', '②', '③', '④'], // O/X형은 ['O', 'X']
  answer: 1,                      // 정답 인덱스 (0-based)
  explanation: '해설 텍스트',
  reference: '근로기준법 제50조',
}
```

`question`은 `innerHTML`로 렌더링되므로 통제된 `<strong>` 강조가 가능하다. 사용자 입력은 들어가지 않는다.

### 카테고리 및 인덱스 매핑

| 인덱스  | 카테고리            | 문항 수 |
|---------|---------------------|---------|
| 0 ~ 4   | 근로시간            | 5       |
| 5 ~ 9   | 연차휴가 및 휴일    | 5       |
| 10 ~ 14 | 임금 및 퇴직금      | 5       |
| 15 ~ 19 | 근로계약 및 취업규칙 | 5      |

### 현재 type 분포
- `case` 사용 문항은 **Q4(야간근로 가산수당 계산)** 1문항뿐. 나머지는 모두 `multiple_choice` 또는 `ox`.
- `case`는 "상황 + 질문" 분리가 가능한 문항에만 사용한다(렌더링 시 노란 상황 박스 표시).

## 핵심 기능 요구사항

### 사용자 식별 (사번)
- 시작 화면에서 **이름 + 사번**을 함께 입력받는다(동명이인 구분).
- 두 필드 모두 비어있지 않아야 시작 버튼 활성화.
- 사번이 Storage 키 식별자이므로, 동일 사번 재응시는 누적, 동일 이름·다른 사번은 별도 기록으로 처리.

### 퀴즈 진행 방식
- 카테고리 선택 없이 **4개 카테고리 전체 20문항을 연속으로** 진행
- 카테고리 순서 고정: 근로시간 → 연차휴가 및 휴일 → 임금 및 퇴직금 → 근로계약 및 취업규칙

### 카테고리 스텝 인디케이터 (문항 화면 상단 sticky)
- 4개 카테고리를 가로로 나란히 표시
- 상태 구분: **완료** (체크 아이콘) / **진행 중** (링 강조 + 카테고리명) / **미진행** (회색)
- `currentQ` 값으로 자동 계산. 모바일에서는 진행 중 카테고리만 이름 노출, 나머지는 아이콘만 표시.

### 진행률 이중 표시
- 카테고리 내 번호: 예) `근로시간 2 / 5`
- 전체 번호 + 프로그레스 바: 예) `전체 7 / 20`

### 피드백 (제출 즉시, 인라인 표시)
- 정답 보기: 초록색 + 체크 아이콘
- 오답으로 선택한 보기: 빨간색 + X 아이콘 (정답 보기도 동시에 초록색으로 표시)
- 나머지 보기: `opacity-60` 흐리게 + 클릭 비활성화
- 결과 배너 + 해설 텍스트 + 법 조항 뱃지 표시

### 결과 화면
- 총점, 정답률, 점수 구간 메시지
  - 18~20점: 🏆 "우수! 근로기준법 전문가입니다."
  - 14~17점: 👍 "양호! 핵심 내용을 잘 이해하고 있습니다."
  - 10~13점: 📚 "보통. 취약 영역을 다시 확인해보세요."
  - 0~9점:  💪 "분발! 근로기준법을 처음부터 학습해보세요."
- 카테고리별 정답수 + 취약 영역 강조 (최저 정답수 카테고리 빨간색 + "취약" 뱃지, 모든 카테고리 점수가 동일하면 표시 생략)
- 오답 목록 (접기/펼치기 토글, chevron 회전)
- 하단 버튼: [다시 도전하기] / [순위 보기]
- 상단에 사용자 이름 + 사번을 함께 표기

### 순위 및 Storage
- `window.storage` API 사용, 모든 호출은 반드시 **`try-catch`로 감싸기** (실패 시 무시하고 로컬 결과만 표시).
- **키는 사번 기반**:
  - 리더보드 키: `"leaderboard:{사번}"` → `shared: true`
  - 개인 기록 키: `"user:{사번}"` → `shared: false`
- 저장 구조:
```javascript
// leaderboard:{사번} (shared: true)
{
  name,             // 최신 이름 (이름 변경 시 갱신)
  employeeId,       // 사번
  bestScore,
  firstAchievedAt,  // bestScore 최초 달성일 (ISO) — 동점 정렬용
  totalAttempts,
  lastDate,         // 마지막 응시일 (ISO)
}

// user:{사번} (shared: false)
{
  name,
  employeeId,
  bestScore,
  records: [{ date, score, total: 20, wrongIds, categoryScores }],
}
```
- 동일 사번 재응시: `records`에 추가, `bestScore`는 최고값으로 갱신, `firstAchievedAt`는 best가 갱신될 때만 갱신.
- 정렬: `bestScore` 내림차순 → 동점 시 `firstAchievedAt` 오름차순(먼저 달성한 사람이 위).
- 본인 식별·내 순위 계산은 모두 **사번 기준**.
- 리더보드 표는 모바일 480px 폭에 맞춰 이름 셀 한 칸에 이름과 사번을 위/아래로 묶어 표시.
- 1~3위 🥇🥈🥉 표시, 본인 행 노란색 배경 + `나` 뱃지.
- 내 기록 요약: 현재 순위 / 최고 점수 / 총 응시 횟수 / 직전 회차 대비 점수 변화(`📈` `📉` `➡️` `✨`).
- 사용자 이름은 표 출력 시 `escapeHtml`로 이스케이프 (XSS 방지).
- 저장 중 풀스크린 로딩 오버레이(스피너) 표시.

## 퀴즈 문항 작성 지침

### 핵심 원칙
- 모든 문항은 **하나의 명확한 정답**만 존재해야 한다
- 법률 비전문가(일반 직원)가 읽어도 문장 의미를 바로 이해할 수 있어야 한다
- 문항과 보기를 합산한 총 글자 수는 **200자 이내**를 원칙으로 한다

---

### 문항 텍스트 작성 규칙

**조건을 구체적으로 명시한다**
- ❌ "장기간 근무한 근로자의 연차는 몇 일인가?"
- ✅ "입사 후 3년 이상 근무한 근로자의 연차 유급휴가는 최대 몇 일인가?"

**측정 단위와 기준을 항상 포함한다**
- ❌ "법정 근로시간은 얼마인가?"
- ✅ "근로기준법상 성인 근로자의 1주 법정 근로시간은 몇 시간인가?"

**부정형 문항은 굵게 강조하거나 괄호로 표시한다**
- ❌ "다음 중 퇴직금 지급 대상이 아닌 경우는?"
- ✅ "다음 중 퇴직금 지급 대상이 **아닌** 것은?"
- 구현: `question` 필드에 `<strong class="text-red-600">아닌</strong>` 형태로 직접 삽입

**사례형 문항은 상황과 질문을 분리한다**
- `type: 'case'`로 지정하면 화면에서 노란 상황 박스 + 질문이 자동 분리된다
- 분리 로직은 `^(.*?[.!])\s*([^.!?]+\?[^?]*)$` 정규식 매칭이므로, "마침표/느낌표로 끝나는 상황 + 물음표로 끝나는 질문" 형식으로 작성한다

---

### 보기 작성 규칙

**보기의 형식과 길이를 통일한다**
- ❌ ① 1년 ② 만으로 2년이 경과한 시점 ③ 36개월 ④ 3년차 도달 시
- ✅ ① 1년 ② 2년 ③ 3년 ④ 4년

**단위를 통일한다**
- ❌ ① 30분 ② 45분 ③ 1시간 ④ 1시간 30분
- ✅ ① 30분 ② 45분 ③ 60분 ④ 90분

**보기 간 겹치거나 포함 관계가 없어야 한다**
- ❌ ① 3일 이내 ② 5일 이내 ③ 3~5일 사이 ④ 1주일 이내
- ✅ ① 3일 ② 5일 ③ 7일 ④ 14일

**O/X 문항은 단일 사실만 묻는다 (복합 조건 금지)**
- ❌ "1년 이상 근무하고 80% 이상 출근한 경우 연차가 발생하며, 미사용 시 수당으로 지급된다" (두 가지 사실 혼합)
- ✅ "1년간 80% 이상 출근한 근로자에게는 15일의 유급휴가가 발생한다"

---

### 해설 작성 규칙

**해설은 다음 3가지를 순서대로 포함한다**
1. 정답이 맞는 이유 (1~2문장)
2. 오답이 틀린 이유 또는 혼동하기 쉬운 포인트 (1문장, 선택)
3. 관련 법 조항 (`reference` 필드에 별도 표시)

**수치가 등장하면 계산 근거를 함께 제시한다**
- ❌ "퇴직금은 평균임금의 30일분입니다."
- ✅ "퇴직금은 계속근로기간 1년에 대해 30일분 이상의 평균임금으로 계산합니다. 예) 평균임금 일 100,000원, 5년 근무 → 100,000원 × 30일 × 5년 = 15,000,000원."

---

### 문항 검수 체크리스트
문항을 작성한 후 아래 항목을 확인한다.

- [ ] 정답이 오직 1개인가
- [ ] 조건(기간·인원·단위)이 구체적으로 명시되어 있는가
- [ ] 보기의 형식·길이·단위가 균일한가
- [ ] 부정형 표현이 있다면 `<strong>` 강조 처리되어 있는가
- [ ] O/X 문항이 복합 조건을 묻고 있지 않은가
- [ ] 해설에 정답 이유와 법 조항이 포함되어 있는가
